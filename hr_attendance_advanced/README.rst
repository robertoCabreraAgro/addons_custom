HR Attendance Advanced
=======================

This module adds the following features:

1. **Option to activate or deactivate the native setting that counts pre-work time as extra hours**.
----------------------------------------------------------------------------------------------------

Odoo natively counts the pre work time as extra hours. This module allows the option to activate or deactivate these native settings.


Configuration
~~~~~~~~~~~~~

Activate the option to not count the pre work time as extra hours.

.. image:: hr_attendance_advanced/static/img/configuration.png
    :width: 600pt



2. **Automatic generation of absences for employees who are late, based on a configured tolerance**.
----------------------------------------------------------------------------------------------------

This feature allows for the automatic creation of absences for employees who are late beyond a defined tolerance.

How the Cron Works:
~~~~~~~~~~~~~~~~~~~

The cron job in this module runs periodically to check employee attendance and automatically generate absences in case of delays. It is scheduled through a schedule activity in Odoo, which is configured to run everyday at 8:00am UTC.

The cron job performs the following steps:

1. **Attendance Check**: It loops through all attendance records for each employee one day before the cron runs.
2. **Delay Calculation**: It compares the `check_in` time with the employee's scheduled start time. If the employee is late, it checks if the delay exceeds the configured tolerance.
3. **Absence Creation**: If the delay exceeds the tolerance, an absence request is automatically created for the employee.

Validations Before Generating Absences:
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Before generating an absence, the system performs the following checks:

1. **Attendance Record Exists**: The system checks if the employee has an attendance record for the specific day. If there’s no `check_in` time, no absence is generated.
2. **Employee Time Zone**: It converts the `check_in` time to the employee's time zone to properly compare it against their work schedule.
3. **Tolerance Check**: It verifies if the employee's delay exceeds the configured tolerance (`tolerance_check_in`).
4. **Absence Already Exists**: The system checks if there’s already an existing absence for the same time period. If one exists, no new absence will be generated.
5. **Workday Check**: It ensures that the attendance record falls on a working day according to the employee's calendar. Non-working days won't generate absences.

What to Do if an Absence Is Invalid:
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

If an absence is invalid, here’s how to handle it:

1. **Accidental Absence Generation**: If an absence was generated by mistake (e.g., due to a cron issue or incorrect delay calculation), it can be manually refused from the absence record.
   - The system doesnt allow to delete absences generated by this cron.

2. **Time Zone**: Make sure each employee has their time zone configured in its profile.
